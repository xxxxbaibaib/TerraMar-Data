# PRD｜个人中心（账户内）v1 草案

文档版本：v1.0（草案）  
关联文档：[PRD_Shanhaiyun_Auth_Membership_v1.md](./PRD_Shanhaiyun_Auth_Membership_v1.md) · [API_Shanhaiyun_User_Membership_contract_v0.md](./API_Shanhaiyun_User_Membership_contract_v0.md) · [PRD_TerraMar_Web_MVP.md](./PRD_TerraMar_Web_MVP.md)  
前置说明：当前官网已实现登录/注册与「我的账户」演示页（`/account`，会员档案、主身份、分轨升级演示）。本文定义登录后 **个人中心 / 个人信息** 的 **信息架构（IA）**、**课程订单状态口径**、各功能模块职责及 **演示 vs 正式** 边界；**不替代** 既有认证 PRD 中的用户模型与契约。  
UI 结构可参考常见电商「个人中心」分区（顶部订单态 + 分组列表），**语义对齐自然教育与公益/公民科学场景**；视觉须沿用 TerraMar 品牌色与现有 `card` / `container-page` 体系，禁止照搬第三方配色。

## 2. 用户与入口

| 状态 | 行为 |
|------|------|
| 未登录 | 顶栏「账户」或直达 `/account` → 展示去登录/注册；`next` 回跳策略与加入流一致（见 `src/lib/joinRouting.ts`）。 |
| 已登录 | 顶栏「账户」→ **个人中心 Hub**（见下节推荐路由）；展示用户摘要（昵称、主身份、会员类型）。 |

## 3. 推荐路由与信息架构

### 3.1 路由建议（实现期择优）

| 路径 | 用途 |
|------|------|
| `/account` | **个人中心 Hub**：订单五态入口、快捷入口列表、摘要卡片；可与现有会员/演示区块共存或折叠为子区。 |
| `/account/orders` | 我的订单列表（按状态筛选）；P1 可为占位页。 |
| `/account/orders/:id` | 订单详情（课程场次、支付、履约节点）；P2。 |
| `/account/footprint` | 我的轨迹（时间线）；P1 占位 + 本地 mock。 |
| `/account/cart` | 课程意向单 / 收藏待报；P1 占位。 |
| `/account/tasks` | 任务中心（升级任务）；与现有分轨等级强关联，可从 Hub 深链。 |
| `/account/addresses` | 收货地址列表与编辑；P1 占位。 |
| `/account/support` | 客服与帮助；P1 外链或静态页。 |
| `/account/profile` | 个人信息（头像、昵称、绑定手机/邮箱等）；P2 对接档案接口。 |
| `/account/security` | 账号与安全（密码、设备、隐私同意、注销）；演示环境须重复密码安全提示。 |

**说明**：若短期不拆路由，允许 Hub 单页内用锚区或折叠面板承载上述模块，但 PRD 以 **可独立成页的模块边界** 为准，便于后续迭代。

### 3.2 Hub 页面区块顺序（自上而下）

1. 页头：标题「个人中心」；可选消息/通知入口（P2）。  
2. **我的订单（课程）**：五状态快捷入口 +「查看全部订单」。  
3. **分组列表**（卡片 1）：我的轨迹、购物车（意向单）、任务中心、赠品/权益（可选，与运营活动挂钩时启用）。  
4. **分组列表**（卡片 2）：收货地址。  
5. **分组列表**（卡片 3）：客服聊天、个人信息、账号与安全。  
6. 底部：退出登录（与现有行为一致）。

## 4. 我的订单（课程）— 状态定义

订单聚合 **自然教育课程/营期** 的交易与履约，**不**按实物电商字面理解；对外文案可统一为「课程订单」。

| 状态 slug | 展示名 | 业务含义（课程域） | 典型下一动作 |
|-----------|--------|-------------------|--------------|
| `pending_payment` | 待付款 | 已创建报名/订单，未完成支付（含定金场景）。 | 去支付、取消订单。 |
| `pending_fulfillment` | 待发货 / 待发资料 | 已支付，待发放 **行前资料 / 电子票据 / 物料包寄送单号**（无实物时可等价为「待发放学习资料」）。 | 查看进度、联系客服。 |
| `pending_receipt` | 待收货 / 待开始 | 已具备参与条件，待 **开营签到 / 线下集合** 或用户确认「已收到资料」；可按产品二选一或并存子状态（文档层允许扩展子状态，MVP 可合并展示）。 | 签到、查看课表。 |
| `to_review` | 评价 | 营期已结束且履约完成，待用户评价 / NPS。 | 去评价。 |
| `refund_after_sale` | 退款/售后 | 退改申请中、已退款、售后处理中。 | 查看售后单、联系客服。 |

**与山海云**：正式环境以山海云订单服务为准；状态迁移、支付回调、发票等 **非官网权威**。官网展示列表来自接口或 BFF 聚合。

**空状态**：无订单时五宫格灰置或展示「暂无课程订单」+ CTA「探索活动」链至 `/programs`。

## 5. 各模块产品说明

### 5.1 我的轨迹

- **定义**：用户在官网内的 **关键行为时间线**（浏览活动、报名、支付、签到、物种记录提交、志愿/公民科学登记、任务完成等）。  
- **演示期**：可由本地事件表或简化 mock 生成；不与生产用户隐私冲突。  
- **正式期**：由山海云审计日志或 `contribution_events` 同类事件流聚合（参见 API 契约中的事件类型扩展思路）。

### 5.2 购物车（课程意向单）

- **定义**：用户 **收藏 / 待付款意向** 的课程列表，与电商购物车 **交互同构**（增删、去结算），文案为教育场景（「意向单」「待报名」）。  
- **衔接**：与活动详情页 `LeadForm` / 未来报名 SKU 打通；演示期可复用 `programs` mock 的 slug 列表。

### 5.3 任务中心（升级任务）

- **定义**：将「升级」游戏化为可完成任务（完成 N 次课程学习、累计志愿时长、提交物种记录等），展示进度条、奖励（等级、徽章、优惠券等运营配置）。  
- **与现有能力对齐**：当前官网演示已有 **主身份 + 分轨等级**（`levelPolicy.ts`、`/account` 演示操作）；任务中心为 **呈现层 + 任务定义表**，长期应对齐 `contribution_events` 与等级重算策略（见会员 API 契约）。  
- **机构账号**：可展示机构维度任务或隐藏个人升级任务（与 PRD 机构单档一致）。

### 5.4 收货地址

- **用途**：教材、物料、证书等 **实物寄送**；一条订单可关联地址快照。  
- **权限**：个人会员默认展示；**机构会员** 可弱化为「机构对接地址」或隐藏个人收货（避免与机构档案冲突），具体以后台配置为准。  
- **演示期**：列表 + 编辑表单占位，数据仅存本地或 mock。

### 5.5 客服聊天

- **MVP**：「联系客服」入口（电话 / 微信 / 企业微信二维码 / 工单链接）+ 服务时间说明。  
- **扩展**：嵌入 IM SDK 或跳转小程序客服；需法务审核留存与隐私条款。

### 5.6 个人信息

- 字段建议：头像、昵称、性别（可选）、简介、绑定手机、绑定邮箱。  
- **数据源**：山海云 `user_profiles`；官网仅编辑入口与校验，**写接口成功后刷新会话缓存**。

### 5.7 账号与安全

- 功能建议：修改密码、登录设备管理、隐私政策与授权撤回、账号注销申请。  
- **演示环境**：须醒目标注「密码仅存本机、禁止生产使用」；正式环境对接 OIDC/短信验证等（见认证 PRD）。

## 6. 交互与 UI / UX 约束（对齐 UI/UX Pro Max 工作流）

以下约束供设计与前端实现共用；技能参考：`frontend/skill/ui_skill/ui-ux-pro-max/ui-ux-pro-max/SKILL.md`。

| 维度 | 要求 |
|------|------|
| 产品类型 | 教育 / 服务类 **SaaS 个人中心**，避免电商误导文案（订单五态已课程化）。 |
| 布局 | **移动端优先**：单列卡片；桌面端 `max-width` 与现有 `tm-page` / `card` 一致，区块间留白充足。 |
| 组件模式 | 订单区：**标题行 +「查看全部」** + **五宫格图标+短文案**；下方为 **分组列表行**（左图标 + 标题 + 右侧 chevron）。 |
| 无障碍 | 列表行可键盘聚焦；图标 `aria-hidden` 或配 `aria-label`；订单状态 **不单独依赖颜色** 传达（配文案/图标）。 |
| 状态反馈 | 每模块具备 **加载中 / 空状态 / 错误** 的默认文案与重试入口。 |
| 动效 | 过渡克制，避免干扰阅读（与 UX 域「animation」最佳实践一致）。 |

## 9. 验收标准（可测）

1. 已登录用户从顶栏「账户」进入后，能在 **一个屏幕内** 理解「订单五态 + 常用功能入口」；未登录访问受保护路径时引导登录且支持 `next` 回跳。  
2. 文档中五类订单状态均有 **课程域释义**，开发与运营可使用同一套 slug。  
3. 「购物车」在对外文案上 **不出现** 纯电商歧义（至少副标题说明为课程意向）。  
4. 任务中心与会员等级逻辑 **不与** [PRD_Shanhaiyun_Auth_Membership_v1.md](./PRD_Shanhaiyun_Auth_Membership_v1.md) 矛盾；扩展事件类型时与 [API 契约](./API_Shanhaiyun_User_Membership_contract_v0.md) 可追踪。  
5. 无障碍抽检：订单区与列表区至少各一处可用键盘完成跳转。  
6. `npm run build` 不因文档变更受影响（纯文档）；实现阶段另立开发与测试清单。
